# 我的 AI 笔记系统

我现在这套 AI 笔记系统，最让我满意的地方其实不是功能多，而是两个笔记搭起来特别丝滑。

Obsidian 和 Get 笔记，我没有把它们当成替代关系，而是直接把它们拆成了两种角色。

一个负责沉淀，一个负责收集。

这样一分，整个系统就顺了。

## 指标卡

- 本地沉淀：Obsidian
- 云端收集：Get 笔记
- 中间层：OpenClaw + Codex

## 为什么沉淀端我用 Obsidian

我很多真正要留下来的东西，都会放在 Obsidian 里。

比如个人资料、长期想法、项目管理、学习资料、书籍笔记，这些都不是随手记一下就完事的东西，而是以后还要反复回来看、反复继续长的内容。

Obsidian 很适合干这个。

第一，它本身就是 Markdown，编辑很轻，也不容易被平台锁死。

第二，它官方同步做得不错，多端使用也方便。

第三，它对 Agent 很友好。文件结构清楚，格式稳定，再配合 Obsidian CLI，像 OpenClaw 或 Codex 去读、去改、去整理，都会很顺。

所以在我这里，Obsidian 不是拿来负责“乱七八糟先收进来”，而是负责长期沉淀。

## 为什么收集端我用 Get 笔记

因为日常收集和长期沉淀，本来就不是一回事。

我平时真正高频遇到的，往往不是一篇完整笔记，而是一堆碎东西：

一个链接，一张图，一段视频，一条音频，一个临时想法，一次会议记录。

这些东西如果全都直接塞进 Obsidian，最后本地库很容易越来越乱。

所以我把这部分交给 Get 笔记。

Get 笔记最适合我的地方，不是“它也是笔记软件”，而是它特别适合做收集端。

它对国内内容的适配很好，很多链接、图片、视频、音频都能识别，而且总结得也准。整个收集过程很丝滑，几乎不会打断我当下的动作。

这点特别重要。

因为收集这个动作，本来就应该越轻越好。你不是每次都想打开本地库、建分类、想结构。很多时候你只是想先接住，别丢。

Get 笔记刚好把这件事做得很好。

## 为什么不用 Agent 直接硬做收集

这也是我后来想明白的一点。

不是所有事情都值得自己重新造一遍。

如果让我直接用 Agent 去处理图片、音频、视频、网页链接、各个平台内容，我就得面对解析、接口、反扒、代理这些一堆脏活。链路会很长，也很容易崩。

而且就算能做出来，也未必划算。

Get 笔记本身已经把这一层做得很好了，它还有自己的 AI，识别和整理都针对这个场景做过优化。那我没必要再拿自己的 Agent 和 token 去重复造一遍轮子。

所以我干脆把“灵感收集”和“日常收集”这一层直接交给 Get 笔记。

## OpenClaw 和 Codex，负责把两边接起来

如果只有 Obsidian 和 Get 笔记，那还只是两个仓库。

真正让这套系统活起来的，是中间这层 Agent。

我现在中间层用的是 OpenClaw 和 Codex。

它们的作用，不是替代这两个笔记工具，而是把两边接起来。

Get 笔记官方本身就支持 OpenClaw 的技能，也提供了 API，所以 Agent 去调用它会很顺。这就意味着，Agent 可以帮我读 Get 笔记里的内容，做第一轮整理、筛选、归类，再把真正有价值的部分往 Obsidian 那边沉淀。

这样一来，这套系统就形成了很清楚的分工：

- Get 笔记负责接住日常碎片
- OpenClaw 和 Codex 负责整理和搬运
- Obsidian 负责长期沉淀

这也是我最喜欢它的地方。

不是因为它看起来高级，而是因为它真的顺。

## 这套系统真正顺的地方

我后来越来越觉得，笔记系统最怕的不是工具不够强，而是什么都想塞进一个地方。

一旦你把收集和沉淀拆开，很多事情就简单了。

Get 笔记像前台，负责把世界先接住。

Obsidian 像后台，负责把真正重要的东西留下。

OpenClaw 和 Codex 则负责在中间把两边接起来。

所以我现在对这套系统的定义，其实不是“一个 AI 笔记工具”。

它更像一个分工明确的笔记工作流。

而它最舒服的地方，就是搭起来之后特别丝滑。
